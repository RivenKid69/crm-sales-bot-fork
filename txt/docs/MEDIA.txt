# Media

## 1. Статус

В основной чат-бот добавлена поддержка медиа-вложений во входящем сообщении.

Сейчас в runtime чата поддерживаются:
- изображения
- аудио / голосовые сообщения
- документы
- видео

Честный текущий статус по проверке:
- изображения: реальный E2E прогоняли, работает
- аудио: реальный E2E прогоняли, работает
- документы: `txt/docx/html/pdf/xlsx/pptx` закрыты кодом; unit/regression прогоны есть, real E2E по офисным документам ещё не гоняли
- видео: кодовый путь уже есть, но отдельный real E2E ещё не прогоняли

## 2. Где это подключено

Основная интеграция находится в:
- `src/api.py`
- `src/media_preprocessor.py`
- `src/llm.py`

Схема работы такая:
1. API принимает `message.attachments`.
2. До вызова `SalesBot.process(...)` запускается `prepare_incoming_message(...)`.
3. `MediaPreprocessor` превращает вложения в компактный текстовый контекст.
4. Уже этот текст подаётся в основной диалоговый pipeline.

То есть для бота медиа не является отдельной веткой логики. Медиа сначала конвертируется в текст, дальше бот работает как обычно.

## 3. Какой формат запроса ожидается

Во входящем API-сообщении можно передавать:
- `type`
- `mime_type`
- `file_name`
- `url`
- `data_base64`
- `text_content`
- `caption`

Минимально это выглядит так:

```json
{
  "session_id": "sess-1",
  "user_id": "user-1",
  "message": {
    "text": "посмотрите, пожалуйста",
    "attachments": [
      {
        "type": "image",
        "mime_type": "image/png",
        "file_name": "photo.png",
        "data_base64": "..."
      }
    ]
  }
}
```

Допускается и attachment-only сценарий, когда `message.text` пустой, но есть вложение.

## 4. Что умеет каждый тип вложения

### 4.1. Изображения

Поддерживаемые типы:
- `image/*`
- расширения: `jpg`, `jpeg`, `png`, `webp`, `gif`, `bmp`

Как это работает:
1. вложение определяется как `image`
2. бинарные данные передаются в мультимодальный LLM
3. LLM возвращает краткое описание изображения, видимый текст и предполагаемый запрос клиента
4. это описание добавляется в подготовленное сообщение

Что уже проверено:
- реальный E2E с внешними image URL
- локальный прогон `png`, `jpg`, `jpeg`, `webp`, `gif`, `bmp`

Нюанс:
- для `gif` сам pipeline не ломается, но в одном из live-прогонов vision-анализ ушёл в fallback; transport и обработка есть, но надёжность авторазбора `gif` ниже, чем у обычных изображений

## 5. Аудио / голосовые сообщения

Поддерживаемые типы:
- `audio/*`
- типы вложений: `audio`, `voice`, `voice_note`, `voice_message`, `audio_message`
- расширения: `mp3`, `wav`, `ogg`, `oga`, `opus`, `m4a`, `aac`, `flac`

Как это работает:
1. вложение определяется как `audio`
2. бинарные данные пишутся во временный файл
3. STT поднимается через `faster-whisper`
4. используется модель `large-v3-turbo`
5. из аудио извлекается текстовая расшифровка
6. расшифровка добавляется в prepared message как обычный текст клиента
7. дальше бот отвечает уже по смыслу голосового

Текущее STT-подключение:
- backend: `faster-whisper`
- модель по умолчанию: `large-v3-turbo`
- язык по умолчанию: `ru`

Управляется через env:
- `MEDIA_STT_MODEL`
- `MEDIA_STT_DEVICE`
- `MEDIA_STT_COMPUTE_TYPE`
- `MEDIA_STT_LANGUAGE`
- `MEDIA_STT_BEAM_SIZE`

Что уже проверено:
- живой E2E на реальном русскоязычном `ogg`-файле из интернета
- бот успешно получил transcript и ответил по содержанию голосового сообщения

Важно:
- для работы аудио в runtime должен быть установлен `faster-whisper`
- веса `large-v3-turbo` должны быть доступны локально или скачиваемы на первом запуске
- если нужен GPU, лучше явно задавать `MEDIA_STT_DEVICE=cuda`
- для уже собранного runtime-образа нужен rebuild / redeploy, чтобы новая STT-зависимость реально попала в окружение

## 6. Видео

Поддерживаемые типы:
- `video/*`
- расширения: `mp4`, `mov`, `avi`, `mkv`, `webm`, `m4v`

Как это работает:
1. вложение определяется как `video`
2. через `ffmpeg` извлекаются ключевые кадры
3. извлечённые кадры отправляются в мультимодальный LLM как набор изображений
4. LLM возвращает краткое описание того, что показано в видео и какой может быть запрос клиента

Текущий статус:
- код реализован
- отдельный real E2E прогон именно для видео ещё не делали

Важно:
- для видео в runtime нужен установленный `ffmpeg`
- без `ffmpeg` видео-анализ работать не будет
- если текущий продовый контейнер собран из старого образа без `ffmpeg`, видео там фактически работать не будет, пока образ не обновят

## 7. Документы

Хотя основной фокус текущего расширения был на аудио / фото / видео, media pipeline также умеет работать с документами.

Поддерживаются:
- `txt`, `md`, `csv`, `json`, `xml`
- `tsv`, `yaml`, `yml`
- `html`, `htm`
- `docx`
- `xlsx`, `xlsm`, `xltx`, `xltm`
- `xls`
- `pptx`, `pptm`, `ppsx`, `potx`, `potm`
- `odt`, `ods`, `odp`
- `pdf`
- `rtf`
- `doc`

Как это работает:
- текстовые файлы декодируются напрямую
- `html` сначала очищается в plain text
- `docx` читается из `word/document.xml`
- `xlsx/xlsm/...` разбираются напрямую из OOXML-архива, с сохранением структуры листов и строк
- `xls` извлекается через `xlrd`
- `pptx/pptm/...` разбираются напрямую из OOXML-архива, с извлечением текста со слайдов
- `odt/ods/odp` читаются через `content.xml`
- `pdf` извлекается через `pypdf` / `PyPDF2` или `pdftotext`, если backend доступен
- `rtf` очищается в текст через встроенный parser
- `doc` извлекается через системный backend `antiword` или `catdoc`
- дальше текст документа кратко суммаризируется LLM

Operational status:
- `docx` работает без внешних системных утилит
- `doc` требует контейнер / окружение, где доступны `antiword` или `catdoc`
- в Dockerfile эти утилиты уже добавлены, но старые контейнеры нужно пересобрать

## 8. Безопасность загрузки по URL

Если attachment приходит как `url`, используется безопасная загрузка:
- только `http/https`
- запрещены URL с credentials
- hostname обязательно должен резолвиться
- non-public IP блокируются
- редиректы проверяются вручную и ограничены
- скачивание идёт потоково
- есть лимит на общий размер вложения

Это сделано для защиты от SSRF и слишком больших загрузок.

## 9. Лимиты и ограничения

Основные runtime-лимиты задаются env-переменными:
- `MEDIA_MAX_ATTACHMENTS`
- `MEDIA_MAX_INLINE_BYTES`
- `MEDIA_DOWNLOAD_TIMEOUT_SECONDS`
- `MEDIA_MAX_DOCUMENT_CHARS`
- `MEDIA_MAX_SUMMARY_CHARS`
- `MEDIA_MAX_VIDEO_FRAMES`
- `MEDIA_MAX_DOWNLOAD_REDIRECTS`

Практический смысл:
- слишком много вложений будут частично скипнуты
- слишком большие файлы будут отклонены
- в бот передаётся не полный сырой контент, а компактная текстовая выжимка

## 10. Что уже подтверждено реальными прогонами

Файлы-артефакты в `results/`:
- `media_real_e2e_20260311_100607.json` — реальные URL-прогоны image/html/pdf
- `media_local_formats_e2e_20260311_103023.json` — локальные прогоны по форматам изображений
- `voice_real_e2e_20260311_112240.json` — реальный E2E на голосовом сообщении

Дополнительно подтверждено unit/regression тестами:
- `xlsx` -> извлечение текста из листов
- `pptx` -> извлечение текста со слайдов

Из этого подтверждено:
- бот понимает изображения и отвечает по смыслу
- бот понимает голосовое сообщение и отвечает по смыслу
- бот умеет использовать документ/HTML/таблицу/презентацию как дополнительный контекст
- видео-путь технически подключён, но ещё ждёт отдельной боевой проверки

## 11. Итог

На текущий момент media support в чате устроен так:
- фото -> мультимодальный разбор -> текстовый контекст -> ответ бота
- аудио -> STT -> текстовый контекст -> ответ бота
- видео -> извлечение кадров -> мультимодальный разбор -> текстовый контекст -> ответ бота
- документы -> извлечение текста / summary -> текстовый контекст -> ответ бота

Итого:
- фото: подключено и проверено
- аудио: подключено и проверено
- видео: подключено технически, но ещё не закрыто отдельным real E2E
